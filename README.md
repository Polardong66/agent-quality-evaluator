# Agent Quality Evaluator (OPI Framework)

高代码 Agent 系统质量评估技能。基于 OPI（Output-Process-Input）三层分析框架，从性能与商业两大维度、六个评估轴对 Agent 系统进行结构化评分，产出可操作的改进路线图。

## 为什么需要它

- Agent 上线前缺少系统化的质量把关标准，评估依赖主观感受
- "回答看起来合理"不等于"Agent 真正完成了任务"——文本通顺无法暴露工具调用失败和复合错误
- 没有可复现的评分体系，不同评审者结论分歧大，改进方向不明确

本技能提供了一套**诊断式评估方法**（用程度/分布/模式替代是非判断），让评估结论可复现、可对比、可追踪。

## 评估框架

### OPI 三层分析

```
Output → Process → Input
看输出结果    看执行过程   看输入处理
准确性+合规   稳定性+速度  可控性+稳定
                  +成本+可控
```

### 六大评估轴（诊断式核心问题）

| 维度 | 评估轴 | 核心问题（诊断式，非是非题） | 通用权重 |
|------|--------|--------------------------|:------:|
| 性能 | **准确性** | 错误集中在哪类任务？引用覆盖率多高？幻觉频率和类型分布？ | 25% |
| 性能 | **稳定性** | 哪些输入条件下容易波动？降级策略覆盖多少异常类型？恢复成功率？ | 25% |
| 性能 | **响应速度** | P50/P95 延迟是多少？用户等待感知在哪个环节最强烈？中间状态是否可审计？ | 15% |
| 商业 | **可控性** | 干预点在决策链的哪个位置？熔断阈值如何设定？回滚粒度到哪一级？ | 10% |
| 商业 | **成本** | 单次任务 Token 分布如何？缓存命中率？重试占总消耗的比例？ | 15% |
| 商业 | **合规性** | 溯源链路覆盖了哪些环节？护栏在检索/工具/输出三层各做了什么？未覆盖盲区在哪？ | 10% |

总分 = Σ(维度得分 × 权重)，维度得分 0-100，总分映射到 A-E 五级。

### 三种权重预设

| 场景 | 适用领域 | 侧重 |
|------|---------|------|
| **通用场景**（默认） | 一般业务 | 准确性 25% + 稳定性 25% |
| **受监管行业** | 金融 / 法律 / 医疗 | 合规性 25%，可控性 15% |
| **实时服务** | ChatBot / 在线客服 | 响应速度 25%，稳定性 25% |

支持对话中自定义权重，调整后即时生效。

## 评估流程

```
步骤1 ──→ 步骤2 ──→ 步骤3 ──→ 步骤4 ──→ 步骤5 ──→ 步骤6
确认权重   收集评分   执行计算   生成报告   改进建议   部署门禁
  🔴         🔴                  🔴                   🔴
```

每步有 🔴 CHECKPOINT，必须用户明确确认后才进入下一步。步骤 1 强制展示权重表并等待「确认」回复，杜绝跳过。

## 核心能力

### 6 步结构化评估
- 权重确认 → 六维诊断式评分 → 脚本自动计算 → 报告生成 → 改进路线图 → 生产就绪门禁

### 5 级成熟度模型

| 等级 | 名称 | 总分 | 可上线 |
|:----:|------|:----:|:------:|
| L1 | 实验原型 | 0-39 | ❌ |
| L2 | 功能验证 | 40-54 | ❌ |
| L3 | 生产就绪 | 55-69 | ⚠️ 内部使用 |
| L4 | 企业级 | 70-84 | ✅ |
| L5 | AI 原生 | 85-100 | ✅ |

### 7 个 APM 管控点
规划控制 / 记忆控制 / 工具控制 / 行动控制 / 编排控制 / 安全控制 / 全流程可观测性

### 组件级深度评估
当某维度 < 60 分时，下沉到路由器、收敛性、工具调用、记忆管理、推理步骤、输出安全六个组件逐项诊断。

### 部署前 9 项门禁
SLO/SLI 定义 / 监控告警 / Runbook / 回滚方案 / 对抗测试 / 安全合规 / 漂移检测 / 模型路由评估 / Kill Switch 配置

### 持续评估飞轮
离线评估（PR/发布前）→ 部署上线 → 在线监控 → 识别退化 → 补充测试用例 → 改进 → 离线评估

### 评估者反例黑名单（11 条）
禁止凭感觉打分、跳过信息不足维度、未确认权重就评分、单次评分就下结论、静态 QA 陷阱、孤立评估 Agent 等 11 种导致评估失真的行为。

### 故障恢复手册（11 条 if-then 分支）
评分超范围 → 提示修正 | 脚本失败 → 手动计算 | 用户中断 → 保存快照 | 权重和不等于 100% → 自动归一化 | 安全漏洞 → 标注阻断项 | 等

### 多 Agent 对比模式
相同权重预设、独立评分、横向对比排名 + 维度差异归因分析

## 快速开始

### 对话触发

在 WorkBuddy 对话中说：

```
帮我评估这个 Agent 的质量
```

Skill 会自动加载，按六步流程引导你完成评估。

### 命令行直接运行

```bash
python3 scripts/evaluate.py \
  --scores '{"accuracy":85,"stability":70,"speed":60,"controllability":75,"cost":80,"compliance":55}' \
  --preset "通用场景"
```

输出示例：

```
============================================================
 Agent 质量评估结果 (OPI Framework)
============================================================
场景: 通用场景
总分: 73.5 / 100  |  等级: B (良好)

  A. 准确性:     85.0 × 25% =  21.25
  B. 稳定性:     70.0 × 25% =  17.50  ← 最大损失
  C. 响应速度:   60.0 × 15% =   9.00  ←
  D. 可控性:     75.0 × 10% =   7.50
  E. 成本:       80.0 × 15% =  12.00
  F. 合规性:     55.0 × 10% =   5.50  ←

改进优先级 (按加权损失分排序):
  P0: 稳定性 (+17.5) — 降级策略覆盖率不足，建议补充异常输入测试
  P1: 响应速度 (+9.0)  — 缺少进度反馈，建议增加步骤级状态推送
  P2: 合规性 (+5.5)   — 来源溯源不完整，建议在工具调用层补充引用链
```

三种预设：`通用场景` / `受监管行业（金融/法律/医疗）` / `实时服务（ChatBot/在线客服）`

### 生成 HTML 报告

```bash
# 评估并生成可视化报告
python3 scripts/evaluate.py --scores '{"accuracy":85,...}' --preset "通用场景" --json-output > result.json
python3 scripts/generate_report.py --input result.json --agent-name "客服Agent v2.3" \
  --one-liner "准确性良好但可控性不足，缺少HITL机制，不建议直接对客上线" -o report.html
```
