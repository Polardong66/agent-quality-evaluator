# Agent 质量评估技能 (OPI Framework)

## 技能用途

对高代码 Agent 系统进行结构化质量评估。基于"Agent训练营第一期资料"中的 OPI
（Output-Process-Input）方法，从性能与商业两大维度、六个评估轴对 Agent 系统进
行系统性评分，并输出改进建议和评估报告。

## 使用场景

- 评估一个已实现的高代码 Agent 系统的质量
- Agent 项目上线前的质量把关评审
- 对比多个 Agent 实现方案的优劣
- 生成结构化的 Agent 质量评估报告

## 核心框架

### OPI 三层分析法

```
评估视角: Output → Process → Input

Output: 看输出结果（准确性、合规性）
Process: 看执行过程（稳定性、响应速度、成本、可控性）
Input: 看输入处理（可控性、稳定性）
```

### 六大评估轴

> **核心问题采用诊断式发问**（程度、分布、模式），避免是非题导致二元评分。每个问题引导评估者挖掘可操作的改进方向，而非简单判断"有/没有"。

| 维度 | 评估轴 | 核心问题（诊断式） | 子维度（2026 业界新增） |
|------|--------|---------|----------------------|
| 性能 | A. 准确性 | 错误集中在哪类任务？引用覆盖率多高？幻觉频率和类型分布？ | 端到端任务完成率、幻觉检测、事实溯源 |
| 性能 | B. 稳定性 | 哪些输入条件下容易波动？降级策略覆盖多少异常类型？恢复成功率？ | 工具使用鲁棒性、环境敏感度、断点恢复 |
| 性能 | C. 响应速度 | P50/P95 延迟是多少？用户等待感知在哪个环节最强烈？中间状态是否可审计？ | 轨迹可审计性、全链路延迟、决策不确定性度量 |
| 商业 | D. 可控性 | 干预点在决策链的哪个位置？熔断阈值如何设定？回滚粒度到哪一级？ | HITL 中断机制、Kill Switch 熔断、权限分级 |
| 商业 | E. 成本 | 单次任务 Token 分布如何？缓存命中率？重试占总消耗的比例？ | 模型路由效率、语义缓存命中率、重试开销 |
| 商业 | F. 合规性 | 溯源链路覆盖了哪些环节？护栏在检索/工具/输出三层各做了什么？未覆盖的盲区在哪？ | 安全护栏嵌入深度、对抗鲁棒性、NIST AI RMF 对齐 |

> **2026 评估核心准则**：抛弃"回答合理即正确"的静态评估方式，必须基于端到端工作流验证 — 仅看输出文本通顺程度无法暴露工具调用失败、复合错误等 Agent 特有故障。

## 执行流程

```
步骤1          步骤2          步骤3        步骤4          步骤5          步骤6
确定场景    →  收集评分    →  执行计算  →  生成报告    →  改进建议    →  部署门禁
权重确认       评分确认                      报告确认                      交叉验证
  🔴              🔴                          🔴              🔴              🔴
```

每个步骤有明确的 🔴 CHECKPOINT，必须通过后才能进入下一步。

### 步骤 1：确定评估场景和权重

**输入**：用户请求评估一个 Agent 系统
**输出**：确认的权重分配表

向用户展示可用权重预设，确认评估场景：

**可用预设**：
- **通用场景**（默认）：准确性和稳定性权重最高
- **受监管行业**（金融/法律/医疗）：合规性和可控性权重提升
- **实时服务**（ChatBot/在线客服）：响应速度和稳定性权重提升
- **自定义**：用户手动指定各维度权重

🔴 **CHECKPOINT — 确认权重**：**强制暂停，不得跳过**。必须做以下三件事，全部完成才能进入步骤 2：

1. 向用户展示完整的权重配置表（六个维度 + 对应权重百分比），格式如下：
   ```
   | 评估轴 | 权重 |
   |--------|:----:|
   | 准确性 | 25%  |
   | 稳定性 | 25%  |
   | 响应速度 | 15%  |
   | 可控性 | 10%  |
   | 成本 | 15%  |
   | 合规性 | 10%  |
   | **合计** | **100%** |
   ```

2. 如果用户未指定场景 → 默认展示**通用场景**权重，并明确告知用户："以下为通用场景默认权重，如需调整请说明。确认无误请回复「确认」。"

3. **等待用户明确回复「确认」或「没问题」后**，才能进入步骤 2。

禁止行为：
- 禁止在用户未确认的情况下自动进入评分
- 禁止假设用户接受默认权重而跳过展示
- 若用户要自定义 → 等待用户给出每个维度的具体权重，验证 Σ=100% 后再次展示确认
- 后续对话中用户说"调整XX权重"即可重新执行本步骤

### 步骤 2：收集评分数据

**输入**：确认的权重分配表 + Agent 系统信息
**输出**：六个维度的 0-100 评分表

对每个维度给出 0-100 的评分。评分依据参考 `references/evaluation_framework.md` 的
「二、详细评分标准」章节。评分方式有两种：

1. **对话评估**：按准确性→稳定性→响应速度→可控性→成本→合规性顺序，逐一讨论各维度表现并打分
2. **直接评分**：用户直接提供各维度分数

**评分示例**（通用场景）：

> 用户说：「我们的 Agent 回答准确但完全没有引用来源，格式有时不一致」
> → 准确性 75（准确但缺引用，对应「良好：有引用但不够完整」）
> → 稳定性 55（格式不一致，对应「较差：经常格式不一致」）
> → 其余维度按对话收集或由用户补充

在评分时参考以下检查要点：

- **准确性**：端到端任务完成率、引用/来源、数据时效、深度均衡、边界场景、幻觉检测、事实溯源
- **稳定性**：工具使用鲁棒性（API/接口异常时自愈能力）、环境敏感度（网络抖动、界面变化下的表现）、格式一致性、异常降级、断点恢复
- **响应速度**：轨迹可审计性（中间步骤是否可追踪）、进度反馈、决策不确定性度量（Output Token Entropy）、全链路延迟
- **可控性**：HITL 中断机制（关键操作前是否有断点）、Kill Switch 熔断、权限分级、输入预检、边界检测
- **成本**：模型路由效率（是否按任务复杂度动态选模型）、语义缓存命中率、重试开销、Token预估、上下文管理
- **合规性**：安全护栏嵌入深度（检索→工具→输出全链路覆盖）、对抗鲁棒性、NIST AI RMF 对齐度、来源溯源

🔴 **CHECKPOINT — 确认评分**：展示六个维度的评分汇总给用户确认。

- 若某个维度信息不足 → 标注「数据不足」，默认给 50 分并告知用户，等待用户修正或确认
- 若用户对某个维度评分有异议 → 重新讨论该维度后修改
- 所有维度评分确认后（无"数据不足"或用户明确接受当前评分），才能进入步骤 3

### 步骤 3：执行评估计算

**输入**：六个维度的评分 + 选定的权重预设
**输出**：加权总分、等级、各维度加权分、APM 管控点评估

使用 `scripts/evaluate.py` 执行计算：

```bash
# 方式1：命令行直接传分数
python3 scripts/evaluate.py \
  --scores '{"accuracy":85,"stability":70,"speed":60,"controllability":75,"cost":80,"compliance":50}' \
  --preset "通用场景"

# 方式2：使用JSON配置文件
python3 scripts/evaluate.py --config eval_config.json

# 方式3：交互模式
python3 scripts/evaluate.py

# 输出JSON格式（便于后续处理）
python3 scripts/evaluate.py --scores '...' --json-output

# 输出报告到文件
python3 scripts/evaluate.py --scores '...' -o report.txt
```

脚本将自动计算：
- 各维度加权得分
- 综合总分与等级（A-E）
- 按加权损失分排序的改进建议
- 五个 APM 管控点状态评估

若脚本执行失败 → 使用手动计算公式：总分 = Σ(维度得分 × 权重)，各维度加权损失分 = (100 - 维度得分) × 权重，参照 `references/evaluation_framework.md`「五、总分计算」的等级表判定等级。手动计算时在报告中标注「手动计算」。

### 步骤 4：生成评估报告

**输入**：评估计算结果（总分、各维度得分、APM评估、改进建议）
**输出**：完整的 HTML/Markdown 评估报告

基于评估结果，使用 `assets/report_template.md` 作为结构模板生成完整的评估报告。

若模板文件不可用 → 直接按以下结构生成报告：

1. **评估总览**：总分、等级、成熟度等级、可上线状态、六维得分一览、一句话结论
2. **各维度诊断分析**：按诊断式核心问题逐项作答（如"错误集中在哪类任务？"）、反模式命中、APM关联
3. **APM 管控点评估**：规划控制/记忆控制/工具控制/行动控制/编排控制/安全控制/全流程可观测性
4. **改进建议与优先级**：集中模块，按 P0/P1/P2 排列，每条链接回对应维度的诊断发现，含当前分、目标分、行动项、预期效果
5. **发展路径规划**：当前状态/短期/中期/长期目标方向
6. **部署就绪门禁**：9 项逐条确认（仅 A/B 级填写）
7. **CI/CD 集成建议**：按当前成熟度等级推荐自动化检查项

输出格式：生成 HTML 报告（便于预览和分享）。若 HTML 生成失败 → 回退为 Markdown 格式。

🔴 **CHECKPOINT — 确认报告**：展示报告概要（总分、等级、Top 3 改进建议）给用户确认。

- **必须等待用户明确回复「确认」后**，才能输出完整报告并进入步骤 5
- 若用户对报告内容满意 → 输出完整报告
- 若用户要求修改 → 根据反馈调整权重或评分后重新计算，再展示

### 步骤 5：提供改进建议

报告中的改进建议应遵循以下原则：

1. **按加权损失分排序**：损失分 = (100 - 得分) × 权重，损失分越高越需要优先关注
2. **针对性建议**：根据具体得分区间给出不同层级的建议
   - 低于60分：列出需要解决的反模式（anti-patterns）
   - 60-75 分：逐项排查检查要点
   - 75-85 分：优化细节
   - ≥85 分：保持即可
3. **可操作**：每个建议必须是具体的、可执行的行动项

### 步骤 6：部署就绪门禁（上线前）

若评估结果显示系统可上线（等级 A 或 B），或用户明确要求上线，执行以下部署就绪确认。

🔴 **CHECKPOINT — 部署前门禁**：对照以下清单逐项确认，全部通过方可建议上线。

借鉴 SRE PRR（生产就绪审查）和 MLOps 质量门禁模式，上线前必须确认：

| # | 门禁项 | 通过标准 | 不通过时的处理 |
|---|--------|---------|--------------|
| 1 | **SLO/SLI 已定义** | 各维度的服务水平目标已量化（如准确性≥85、稳定性≥75） | 要求用户提供目标值，否则标注「目标未定义」 |
| 2 | **监控与告警已配置** | 生产环境有六维度的监控和异常告警机制，含决策不确定性度量 | 在报告中增加「监控缺失」风险提示 |
| 3 | **Runbook 已就绪** | 常见故障（Token失控、格式异常、超时、工具调用失败）有操作手册 | 列出缺失的 Runbook 条目供补全 |
| 4 | **回滚方案已验证** | 新版本出现严重问题时能快速回退到上一稳定版本 | 标注「无回滚方案」为高风险项 |
| 5 | **对抗/异常输入测试已通过** | 系统对提示注入、对抗输入、边界异常输入有防御 | 建议在上线前补充红队测试 |
| 6 | **安全合规已确认** | 行业特定合规要求已满足；安全护栏嵌入检索→工具→输出全链路 | 标注具体的合规缺口 |
| 7 | **漂移检测已启用** | 有输入分布变化、性能衰减、决策不确定性趋势的监控 | 建议部署后前两周每日评估一次 |
| 8 | **模型路由策略已评估** | 是否按任务复杂度动态分发模型（SLM/LLM），简单任务不被过度分配大模型 | 建议引入语义缓存和模型路由以优化成本 |
| 9 | **Kill Switch 已配置** | 高风险操作（数据库写入、外部 API 调用、资金操作）有强制熔断机制 | 标注为「阻断项」，必须在上线前配置 |

🔴 **CHECKPOINT — 交叉验证**：若评估由单人完成，建议引入第二评估者独立打分。

- 若两轮评分差异 > 10 分 → 针对差异维度展开讨论，取讨论后的共识分数
- 若差异 ≤ 10 分 → 取平均值作为最终分数，报告中标注「双人交叉验证」
- 借鉴 IBM Agent 治理框架中的跨职能评审委员会模式：复杂系统建议工程、产品、业务三方代表共同参与评审

## 组件级深度评估（按需启用）

当六维度的整体评估发现某个维度得分 < 60，或用户要求深入诊断时，可启用组件级评估。参考 IBM Agent 治理框架的组件检查模式：

| 组件 | 评估问题 | 对应的六大维度 |
|------|---------|-------------|
| **路由器/意图识别** | 是否正确识别用户意图？是否为给定输入选择了正确的工具/技能？参数提取是否准确？ | 准确性、可控性 |
| **收敛性** | Agent 是否在推进任务而非陷入循环？多次交互中解决步骤是否在减少？是否存在不必要的迭代？ | 成本、响应速度 |
| **工具/函数调用** | API 调用是否准确？参数格式是否正确？工具错误时是否有恢复机制？ | 稳定性、成本 |
| **记忆/上下文管理** | 过往信息是否有效引用？多轮交互中信息是否一致？上下文是否在持续优化而非无限膨胀？ | 稳定性、成本 |
| **推理步骤** | 中间推理步骤是否可审查？步骤间逻辑是否一致？是否存在跳过关键步骤的捷径行为？ | 准确性、可控性 |
| **输出安全** | 生成内容是否在交付前经过安全校验？是否对提示注入有抵抗力？输出是否包含 PII 或敏感信息？ | 合规性、可控性 |

组件级评估结果补充到报告中「APM 管控点评估」章节，对每个组件标注 ✅/⚠️/❌ 状态。

## 权重调整

用户可在对话中随时调整权重。调整方式：

- "把准确性的权重调到 30%"
- "用受监管行业场景重新评估"
- "把合规性权重翻倍"

调整后重新运行评估脚本即可获得更新后的报告。

默认权重配置和场景预设详见 `references/evaluation_framework.md`。

## 故障恢复手册

评估过程中遇到异常时按以下分支处理：

| 触发条件 | 一线修复 | 仍失败时兜底 |
|---------|---------|------------|
| 用户提供的评分超出 0-100 范围 | 提示正确范围并要求修正，不自动截断 | 标注为「数据异常」，该维度不参与计算，报告中单独说明 |
| scripts/evaluate.py 执行失败 | 检查 Python 环境，重新安装依赖包 | 使用手动计算公式，参照 evaluation_framework.md「五、总分计算」 |
| 用户中断评估中途离开 | 保存当前评分快照到临时 JSON 文件 | 下次对话恢复时读取快照，从断点步骤继续 |
| evaluate.py 计算结果与手动计算不一致 | 优先信任脚本，检查手动计算过程 | 若差异 > 1 分 → 报告两者结果并标注差异原因，由用户决定 |
| 用户同时评估多个 Agent（对比模式） | 对每个 Agent 独立执行完整流程 | 最后统一对比：将各 Agent 的 JSON 结果汇总为对比表 |
| 权重总和不为 100% | 自动归一化处理，并在报告中标注「权重已自动归一化」 | 提示用户原始权重并确认是否接受归一化结果 |
| 用户指定的场景预设不存在 | 列出三个可用预设供选择 | 默认使用通用场景预设，标注「预设未识别，已使用默认」 |
| Token 消耗过大导致上下文不够 | 压缩之前步骤的对话（保留关键评分决策） | 转为 --config JSON 文件模式，纯脚本计算 |
| 评估后性能漂移（用户反馈系统表现下降） | 对比历史评估记录，识别退化的维度 | 重新执行完整评估流程，标注「漂移检测评估」 |
| 发现安全漏洞或合规缺口 | 在报告中单独标注为「阻断项」，建议上线前修复 | 若为高风险行业 → 建议执行红队安全测试后再评估 |
| 组件级评估发现深层问题 | 将组件问题映射到对应六维度，评估是否需要降级评分 | 在 APM 管控点评估中补充组件诊断详情 |

### 多 Agent 对比模式

若用户要求对比多个 Agent 方案：

1. 对每个 Agent 分别执行步骤 1-3，保存 JSON 结果
2. 所有 Agent 评分完成后，生成对比表：横向为 Agent 名称，纵向为六维度得分 + 总分 + 等级
3. 使用相同权重预设评估所有 Agent，确保可比性
4. 对比报告额外包含：各维度差异分析、总分差异归因、综合推荐排名

若某 Agent 信息不足无法评估 → 在对比表中标注「数据不足-未评级」，不参与排名。

## 代理能力成熟度模型

参照业界 Agent 生产就绪标准（Level 3+ 为可上线门槛），将评估结果映射到 5 级成熟度：

| 等级 | 名称 | 总分范围 | 特征 | 典型差距 |
|------|------|---------|------|---------|
| **L1** | 实验原型 | 0-39 | 基本功能不可靠，无监控，无错误处理 | 需从架构层面重构 |
| **L2** | 功能验证 | 40-54 | 核心功能可运行，但缺少稳定性保障和边界处理 | 补全异常处理和格式标准化 |
| **L3** | 生产就绪 | 55-69 | 日常场景可用，有基本监控，有故障恢复机制 | 完善可观测性和成本管控 |
| **L4** | 企业级 | 70-84 | 全链路可观测，有 HITL 机制，成本可控，合规达标 | 引入模型路由、持续评估飞轮 |
| **L5** | AI 原生 | 85-100 | 端到端自动化验证，语义缓存，对抗鲁棒，多 Agent 协作治理 | 建立跨 Agent 编排标准和联邦评估 |

每季度重新评估一次，追踪成熟度演进趋势。L3 以下不建议直接面向终端用户。

## 持续评估与 AI-CI/CD 集成

借鉴 2026 年业界实践，将评估嵌入持续交付流程：

### 离线评估（每次 PR / 发布前）

| 检查项 | 方法 | 频率 |
|--------|------|------|
| 回归测试 | 用历史失败案例集验证新版本不引入退化 | 每次 PR |
| 端到端验证 | 在沙箱环境中运行完整工作流，验证工具调用链路 | 每次发布前 |
| 对抗测试 | 用变异输入（API 异常、超长上下文、对抗 prompt）验证鲁棒性 | 每次大版本 |
| 成本回归 | 对比新旧版本在相同任务集上的 Token 消耗 | 每次发布前 |

### 在线评估（部署后持续监控）

| 监控项 | 指标 | 告警阈值 |
|--------|------|---------|
| 性能漂移 | 六维度得分周环比 | 任一维度下降 > 10 分 |
| 决策不确定性 | Output Token Entropy 均值 | 连续 3 天上升趋势 |
| 工具调用失败率 | API 调用错误占比 | > 5% |
| 用户反馈信号 | 显式评分 + 隐式满意度（重复提问率） | 满意度 < 70% |

### 持续评估飞轮

```
离线评估 → 部署上线 → 在线监控 → 识别退化 → 补充测试用例 → 改进 → 离线评估
```

评估报告中建议增加「CI/CD 集成建议」章节，根据当前成熟度等级推荐应优先接入的自动化检查项。

## 重要规则

1. **评分必须基于事实**：不要凭空给分，需结合用户提供的信息或代码分析
2. **改进建议必须具体**：不能只说"需要改进"，要给出具体的行动项和检查点
3. **报告必须完整**：即使某些维度信息不充分，也要标注"数据不足"而非跳过
4. **支持迭代评估**：用户可以在改进后重新评估，对比前后变化

## 评估者反例黑名单 — 禁止行为

以下行为会导致评估结果失真或不可信，评估时必须避免：

| # | 禁止行为 | 原因 | 正确做法 |
|---|---------|------|---------|
| 1 | **凭感觉打分** | 无事实依据的评分不可复现，不同评估者结果不一致 | 每个维度评分必须引用具体的检查要点或反模式作为依据 |
| 2 | **跳过信息不足的维度** | 缺失维度会导致总分失真，无法全面反映系统质量 | 标注「数据不足」并给默认 50 分，明确告知用户此维度需要补充信息 |
| 3 | **未确认权重就评分** | 权重场景不同会显著改变总分，用户可能事后质疑 | 步骤 1 必须通过 CHECKPOINT 确认权重后，才能进入评分 |
| 4 | **单次评分就下结论** | Agent 系统可能因测试输入不同表现差异大 | 建议用户多次评估取平均，或在报告中标明「本次评估基于单次观察」 |
| 5 | **忽略 APM 管控点** | 仅看六维度得分不看管控点，会遗漏过程性质量缺陷 | 报告必须包含五个 APM 管控点评估 |
| 6 | **改进建议过于笼统** | "提高稳定性"类建议无操作性，用户无法执行 | 每条建议必须包含：具体行动、涉及维度、预期效果 |
| 7 | **混淆评估对象** | 把 Agent 系统的质量问题归因到用户输入，或反之 | 明确区分"系统自身问题"和"输入质量问题"，在报告中分别标注 |
| 8 | **不保留评估记录** | 缺乏历史评估记录无法追踪质量变化趋势 | 每次评估完成后保存 JSON 结果文件，供后续对比 |
| 9 | **静态 QA 评估陷阱** | 仅看输出文本通顺就判为正确，忽略工具调用失败、复合错误等 Agent 特有故障 | 必须基于端到端工作流验证，在真实交互环境（或沙箱）中评估 |
| 10 | **孤立评估 Agent** | 忽视编排框架（LangChain/CrewAI/LangGraph 等）对性能的显著影响 | 评估时注明使用的编排框架及版本，区分「框架问题」和「Agent 逻辑问题」 |
| 11 | **忽略模型路由策略** | 不评估 Agent 是否按任务复杂度合理选择模型，导致成本或质量失衡 | 检查是否存在模型路由逻辑，评估简单任务是否被过度分配给大模型 |
