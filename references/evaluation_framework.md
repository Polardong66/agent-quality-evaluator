# Agent 质量评估框架 (OPI Framework)

## 一、框架总览

本框架源自"Agent训练营第一期资料-管控"，采用 **OPI（Output-Process-Input）三层分析法**，从 **性能** 与 **商业** 两大维度，对高代码 Agent 系统进行结构化质量评估。

### 两大维度

| 维度 | 关注点 | 核心问题 |
|------|--------|---------|
| **性能维度** | 系统技术表现 | Agent 做得好不好？稳不稳定？快不快？ |
| **商业维度** | 业务落地能力 | Agent 能否真正投入使用？成本是否可控？是否合规？ |

### 六大评估轴

```
性能维度 (Performance)
├── A. 准确性 (Accuracy)
├── B. 稳定性 (Stability)
└── C. 响应速度 (Response Speed)

商业维度 (Business)
├── D. 可控性 (Controllability)
├── E. 成本 (Cost)
└── F. 合规性 (Compliance)
```

---

## 二、详细评分标准

每个评估轴按 0-100 分打分，乘以权重后计入总分。

### A. 准确性 (Accuracy) — 性能维度

**定义**：Agent 输出的内容是否正确、可信、在边缘场景下依然可靠。

| 分数区间 | 等级 | 标准 |
|---------|------|------|
| 90-100 | 优秀 | 输出信息完全准确，有完整引用来源，数据时效有保障，边界场景处理正确 |
| 75-89 | 良好 | 主要信息准确，偶有小瑕疵，有引用但不够完整或一致 |
| 60-74 | 及格 | 基本正确但缺乏可验证来源，偶有错误或过时信息 |
| 40-59 | 较差 | 存在明显事实错误或过时信息，引用来源严重缺失 |
| 0-39 | 不合格 | 大量错误信息，无引用来源，不可信赖 |

**检查要点**：
- 输出是否包含可追溯的引用/来源链接
- 信息是否反映当前时效（非过时数据）
- 各部分内容深度是否均衡，论据是否充分
- 边界场景/异常输入下的表现
- 是否存在明显的幻觉或编造行为

**典型反模式**：
- 报告/输出没有任何引用和来源标注
- 反思轮数一刀切，不根据主题复杂度自适应调整
- 段落深度和质量参差不齐
- 输出内容缺乏数据时效保障机制

---

### B. 稳定性 (Stability) — 性能维度

**定义**：Agent 在持续运行、高并发、异常输入等场景下能保持一致的预期行为。

| 分数区间 | 等级 | 标准 |
|---------|------|------|
| 90-100 | 优秀 | 格式始终一致，异常输入优雅降级，局部错误不影响整体，支持断点恢复 |
| 75-89 | 良好 | 大部分场景稳定，偶有格式波动，异常场景有基本处理但不够完善 |
| 60-74 | 及格 | 正常场景可用，但异常输入容易导致不稳定输出或部分功能失效 |
| 40-59 | 较差 | 经常出现格式不一致，异常输入导致崩溃，中间结果丢失 |
| 0-39 | 不合格 | 频繁崩溃，输出格式随机，完全不具备生产级稳定性 |

**检查要点**：
- 输出格式是否保持一致（解析性、结构可预期）
- 异常输入下是否优雅降级而非崩溃
- 出错时中间结果是否保留（能否断点恢复）
- 是否存在 Token 超限导致崩溃的问题
- 不同主题/语言/规模的输入下表现是否一致

**典型反模式**：
- 出错直接崩溃，中间结果全部丢失
- 中文输入 Token 超限直接崩溃
- 输出格式不稳定，需要人工二次整理
- 不稳定行为没有监控和告警机制

---

### C. 响应速度 (Response Speed) — 性能维度

**定义**：用户从发出请求到获得结果的全链路等待时间，以及进度可感知性。

| 分数区间 | 等级 | 标准 |
|---------|------|------|
| 90-100 | 优秀 | 响应快速，用户能实时感知任务进展，预期完成时间明确 |
| 75-89 | 良好 | 响应可接受，有基本进度反馈，等待时间可预期 |
| 60-74 | 及格 | 响应较慢但可接受，缺乏进度反馈，用户不知道等待多久 |
| 40-59 | 较差 | 响应缓慢，执行过程完全不透明，用户无法判断是否卡住 |
| 0-39 | 不合格 | 响应极慢或频繁超时，无任何进度指示 |

**检查要点**：
- 是否有执行进度反馈机制
- 用户能否实时感知任务进展
- 是否提供预计完成时间
- 全流程是否可观测（各个环节耗时可见）
- 是否有超时控制和降级策略

**典型反模式**：
- 执行过程完全不透明，不知道跑到哪里
- 没有任何进度指示或中间状态反馈
- 无超时控制导致长时间无响应

---

### D. 可控性 (Controllability) — 商业维度

**定义**：Agent 的行为是否在预期范围内，触及业务边界时是否有干预机制。

| 分数区间 | 等级 | 标准 |
|---------|------|------|
| 90-100 | 优秀 | 行为完全可控，异常输入提前识别，业务边界清晰，有完善的干预/回滚机制 |
| 75-89 | 良好 | 大部分场景可控，有基本边界检测，干预机制存在但不完善 |
| 60-74 | 及格 | 正常场景可控，但边界场景缺乏预检，干预门槛较高 |
| 40-59 | 较差 | 存在明显失控风险，缺乏边界检测和干预手段 |
| 0-39 | 不合格 | 行为完全不可控，无任何保护措施或干预机制 |

**检查要点**：
- 是否对输入进行预检（合法性、规模、领域匹配）
- 非业务相关的输入是否有明确拒绝提示
- 执行过程中是否有"刹车"或人工干预入口
- 是否支持回滚或撤销操作
- 系统行为的可解释性和可审计性

**典型反模式**：
- 用户输入本身带来不确定性（无输入校验）
- 对输入没有任何预检，异常输入直到执行失败才暴露
- 缺乏可中断/可干预的机制

---

### E. 成本 (Cost) — 商业维度

**定义**：每次推理的 Token 消耗和工具调用费用是否在可接受的业务范围内。

| 分数区间 | 等级 | 标准 |
|---------|------|------|
| 90-100 | 优秀 | 成本可精确预估且稳定，有完善的成本管控机制，无浪费 |
| 75-89 | 良好 | 成本基本可预测，有基本管控，偶有波动但在预算内 |
| 60-74 | 及格 | 成本有一定可预测性，但缺乏有效管控手段 |
| 40-59 | 较差 | 成本不可预测，存在明显浪费或失控风险 |
| 0-39 | 不合格 | 成本完全失控，Token 消耗线性增长或指数爆炸 |

**检查要点**：
- 单次任务 Token 消耗是否可预估
- 是否有执行循环终止保护（防止无限循环）
- 是否有上下文管理机制（避免 Token 线性增长）
- 工具调用次数是否合理（是否有冗余调用）
- 成本是否随输入规模线性可预期

**典型反模式**：
- 执行循环没有终止保护，存在 Token 失控风险
- 没有上下文管理机制，Token 消耗线性增长失控
- 反思/重试循环没有上限控制

---

### F. 合规性 (Compliance) — 商业维度

**定义**：Agent 的输出是否满足业务场景的合规要求（尤其金融、法律、医疗等受监管行业）。

| 分数区间 | 等级 | 标准 |
|---------|------|------|
| 90-100 | 优秀 | 完全合规，每个关键结论都有可溯源依据，符合行业监管要求 |
| 75-89 | 良好 | 基本合规，主要结论有依据，少数次要点缺乏溯源 |
| 60-74 | 及格 | 无明显违规，但可追溯性不足 |
| 40-59 | 较差 | 存在合规风险，结论缺乏依据，可能违反行业规范 |
| 0-39 | 不合格 | 严重合规问题，输出可能造成法律/监管风险 |

**检查要点**：
- 每个关键结论是否有可追溯的来源链接
- 内容是否来源于公开/授权信息
- 是否涉及内部敏感数据
- 输出是否包含免责声明（如适用）
- 是否符合特定行业的监管要求

**典型反模式**：
- 报告没有任何引用和来源标注（与准确性相关联）
- 内容来源不明，无法追溯
- 缺乏合规审查机制

## 三、APM 管控点（Agent Process Management）

在评估过程中，需关注以下五个关键管控点：

| 管控点 | 定义 | 评估关注 |
|--------|------|---------|
| 1. 规划控制 (Planning Control) | Agent 如何识别意图、理解任务、拆解步骤 | 任务拆解是否合理？步骤是否冗余？是否有多方案评估？ |
| 2. 记忆控制 (Memory Control) | Agent 保存哪些信息、生命周期管理、上下文完整性与 Token 成本平衡 | 上下文是否切题？重要信息是否遗漏？是否过度保留无关信息？ |
| 3. 工具控制 (Tool Control) | 调用哪些工具、在什么条件下调用、如何处理工具返回、如何从工具错误恢复 | 工具选择是否准确？调用参数是否正确？错误恢复是否合理？ |
| 4. 行动控制 (Action Control) | 输出什么内容、以什么格式输出、输出边界在哪里 | 输出格式是否稳定？内容是否越界？结果是否可用？ |
| 5. 全流程可观测性 (Full-process Observability) | 对整个执行过程的监控能力 | 是否可追踪？各环节耗时和资源消耗是否可见？ |

## 四、默认权重配置

基于原始资料中各评估轴的优先级（★数量）推导的初始权重。可在对话中调整。

### 通用场景（默认）

| 评估轴 | 优先级 | 默认权重 | 满分 |
|--------|--------|---------|------|
| A. 准确性 | ★★★ | 25% | 100 |
| B. 稳定性 | ★★★ | 25% | 100 |
| C. 响应速度 | ★★ | 15% | 100 |
| D. 可控性 | ★★ | 10% | 100 |
| E. 成本 | ★★ | 15% | 100 |
| F. 合规性 | ★★ | 10% | 100 |

**理由**：原始资料中准确性和稳定性被标记为最高优先级（★★★），响应速度、成本为中等（★★），可控性和合规性视场景而定。

### 受监管行业场景（金融/法律/医疗）

| 评估轴 | 调整后权重 | 满分 |
|--------|-----------|------|
| A. 准确性 | 20% | 100 |
| B. 稳定性 | 20% | 100 |
| C. 响应速度 | 10% | 100 |
| D. 可控性 | 15% | 100 |
| E. 成本 | 10% | 100 |
| F. 合规性 | 25% | 100 |

### 实时服务场景（ChatBot / 在线客服）

| 评估轴 | 调整后权重 | 满分 |
|--------|-----------|------|
| A. 准确性 | 20% | 100 |
| B. 稳定性 | 25% | 100 |
| C. 响应速度 | 25% | 100 |
| D. 可控性 | 10% | 100 |
| E. 成本 | 10% | 100 |
| F. 合规性 | 10% | 100 |

## 五、总分计算

```
总分 = Σ (维度得分 × 维度权重)

其中:
- 维度得分 ∈ [0, 100]
- Σ权重 = 1.0
```

### 总分等级

| 总分区间 | 等级 | 含义 |
|---------|------|------|
| 85-100 | A (优秀) | 系统质量高，可直接上线使用 |
| 70-84 | B (良好) | 系统整体可用，建议优化几个短板后上线 |
| 55-69 | C (及格) | 系统基本可用，但存在明显短板需改进 |
| 40-54 | D (较差) | 系统存在严重问题，需大幅改进后重新评估 |
| 0-39 | E (不合格) | 系统质量不达标，需重构 |
